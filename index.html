Storage Profiles

Choose Sync or Async Object Materialization Per Topic

Nereus separates the primary WAL from object materialization. A topic can run with BookKeeper only, synchronous object visibility, or asynchronous object materialization while preserving the same streamId + offset truth.

BOOKKEEPER_WAL_ONLY

WAL only

Primary WAL: BookKeeper

Object materialization: disabled

Ack boundary: BK durable + stable projection

BookKeeper WAL + Objects

BOOKKEEPER_WAL_SYNC_OBJECT

Sync visibility

Primary WAL: BookKeeper

Object materialization: synchronous

Ack boundary: BK durable + Oxia visible/index commit

BOOKKEEPER_WAL_ASYNC_OBJECT

Async object

Primary WAL: BookKeeper

Object materialization: background worker

Ack boundary: BK durable fast ack + stable projection

Object WAL + Objects

OBJECT_WAL_SYNC_OBJECT

Sync visibility

Primary WAL: Object WAL

Object materialization: synchronous

Ack boundary: Object WAL durable + Oxia visible/index commit

OBJECT_WAL_ASYNC_OBJECT

Async object

Primary WAL: Object WAL

Object materialization: background read-optimized objects

Ack boundary: Object WAL durable fast ack + stable projection

StorageProfile.OBJECT_WAL is kept as a legacy Phase 1 alias and canonicalizes to OBJECT_WAL_SYNC_OBJECT. StorageProfile.defaultDurabilityLevel() is the code-level branch point: sync object profiles use WAL_DURABLE_AND_INDEX_COMMITTED; async and WAL-only profiles use WAL_DURABLE.

Core Design

Five Profiles, One Stream Truth

Nereus is no longer a single synchronous-profile design. The current design separates storage profiles, ack boundaries, materialization lag, protocol projection, and future implementation gates.

🔲

Five Canonical Profiles

BK-only, BK sync object, BK async object, Object WAL sync object, and Object WAL async object are explicit API and metadata choices.

🔄

Two Write Modes

Sync-visibility profiles wait for visible/read index publication. Async-materialization profiles fast-ack after primary WAL durability and stable projection.

📐

Unified streamId + offset

Pulsar MessageId, ManagedLedger Position, Kafka offset, cursor progress, compaction, and lakehouse snapshots all project from one internal coordinate.

⚡

Profile-aware Future Gates

Each Future must state how sync, async, BK-only, BookKeeper WAL, and Object WAL paths behave instead of assuming one OBJECT_WAL path.

🔌

Three Integration Paths

Pulsar native with Oxia, Pulsar Broker + KoP with Oxia, and future Kafka native with KRaft all reuse the same storage core.

☑️

Metadata-Driven Correctness

Oxia remains the offset, fencing, manifest, cursor, and visibility authority. Object storage and BookKeeper store bytes, not stream truth.

🗄️

Async Materialization Lag

Async-materialization paths track materialization tasks, checkpoints, lag records, lag bytes, and primary WAL retention protection as first-class operations.

📦

Remote Storage Compatibility

When Nereus object materialization owns a topic, Kafka Remote Log Storage and Pulsar offload must be mapped or rejected to avoid a second remote log.

Implementation Roadmap

8 Futures: Profile-Aware From Core to Protocol Semantics

The storage core comes first, but every future now has to explain both synchronous visibility and asynchronous object materialization behavior.

0

Project Scaffolding & Module Boundaries Done

Standalone Gradle repository, module boundaries, design-doc index, and fork workflow. Phase 0 is complete.

1

Core StreamStorage + Object WAL + BookKeeper WAL Gated

Phase 1/1.5 final-gated. F1-BK BK-M1-M4 final-gated (BK-M5 partial). Protocol-neutral StreamStorage API, Object WAL + BookKeeper WAL, Oxia stream-head commit, offset index, read resolver, profile-aware API.

2

ManagedLedger Facade Gated

Complete/final-gated. 19 scenarios pass. Real dual-broker gate with Oxia, LocalStack S3, stock BookKeeper. Position/bytes stability, hybrid coexistence, unload/reload/restart verified.

3

Cursor / Subscription State Gated

Complete/final-gated. Durable cursor CAS, snapshot bytes, ack-state epochs, owner-session claims, cursor protocol activation, F4 snapshot inventory handoff.

4

Materialization + Compaction + GC Gated

Complete/final-gated (203/203 tasks, 21m47s). Parquet NCP1/NTC1/NRC1, generation replacement, async materialization, GC/retirement, real two-broker/two-worker contention, million-entry recovery.

5

KoP / Kafka Compatibility Future 5

Kafka offset equals stream record offset. Produce/Fetch mapping, group offsets, transaction markers, leader epoch projection, and topic compaction projection reuse the same Nereus storage truth.

6

Lakehouse SBT / SDT Future 6

Stream-Backed Tables, external Stream-Delivered-to-Table flows, Iceberg-first catalog commit, repair state, delivery idempotence, and lag metrics. Catalog commits stay off the producer ack path.

7

Routing / Brown-out / Elasticity Future 7

Broker sessions, zone-aware routing, preferred broker, append session transfer, degraded broker brown-out, cache invalidation, and Pulsar/Kafka routing projection.

8

Advanced Pulsar Semantics Future 8

Key_Shared, delayed delivery, pending ack transactions, replicated subscriptions, schema/system topic bootstrap, topic compaction compatibility, and geo-replication.
